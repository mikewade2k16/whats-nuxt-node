## Regra Multi-tenant (Obrigatoria)

- Toda nova pagina/fluxo com CRUD deve ter ownership por `client_id` desde o inicio.
- Usuario com `user_type = client`: nao pode ver lista de clientes e so enxerga/cria/edita/exclui dados do proprio cliente.
- Usuario admin/manager: pode escolher o cliente alvo na criacao/edicao quando o fluxo exigir.
- Itens internos devem usar o cliente padrao da operacao administrativa (`crow`/empresa) para nao ficarem visiveis para clientes finais.
- Em producao, criacao por usuario `client` deve vincular automaticamente ao `client_id` da sessao/autenticacao (sem depender de input manual).
- Padrao frontend obrigatorio: usar Pinia para contexto de sessao/tenant e usar BFF (`/api/admin/*`) com headers de acesso (`x-user-type`, `x-client-id`) para toda chamada.
- Simulacao de sessao deve existir no admin em dev/teste: alternar `admin/client` e cliente ativo globalmente para validar isolamento por tenant.
- Padrao realtime obrigatorio para CRUD: cada manager deve registrar sua entidade no `useTenantRealtime` para refetch por evento (`created`, `updated`, `deleted`) quando websocket estiver ativo.
- Toda nova pagina deve ser websocket-ready: contratos de dados e stores precisam prever sincronizacao em tempo real (eventos de create/update/delete) sem reescrever a tela.
- Toda nova rota/API do BFF deve consumir `resolveAccessContext` no backend antes de listar/criar/editar/excluir dados tenant.
- Regra obrigatoria de seguranca: esconder no frontend (menu/coluna/filtro) melhora UX, mas NAO substitui autorizacao backend; toda API deve validar role/ownership no servidor.
- Regra obrigatoria para dados sensiveis: endpoints admin-only devem retornar `403` para perfil `client`; endpoints tenant devem retornar somente dados do `client_id` efetivo da sessao.
- Regra obrigatoria para campos sensiveis: nao expor credenciais/senhas/tokens brutos em respostas de listagem.
